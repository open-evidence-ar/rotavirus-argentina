Common pitfalls and solutions specific to this project. Loaded on demand from `AGENTS.md §8`.

## 1. Windows console encoding (cp1252)

**Symptom:** Python script prints `UnicodeEncodeError: 'charmap' codec can't encode character '\u2014'` (or `\u2192`, `\u2248`, `\u00a7`, `\u00d7`).

**Cause:** Default Windows console encoding is cp1252, cannot encode em-dash (—), right-arrow (→), approx (≈), section sign (§), times (×), n-tilde (ñ), o-acute (ó), divide (÷). The report uses all of these.

**Fix (script level):** Add at the top:
```python
import sys
try:
    sys.stdout.reconfigure(encoding="utf-8")
    sys.stderr.reconfigure(encoding="utf-8")
except (AttributeError, ValueError):
    pass
```

**Fix (session level):** Prepend `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8` before invoking Python from PowerShell.

**Files affected:** both `scripts/validate_calculations.py` and `scripts/check_sources.py` already have the fix applied.

## 2. Microsoft Store "python.exe" stub

**Symptom:** Running `python` in PowerShell opens a Microsoft Store prompt instead of executing.

**Cause:** Windows includes a 0-byte stub at `C:\Users\<user>\AppData\Local\Microsoft\WindowsApps\python.exe` that forwards to the Store.

**Fix:** Always use the explicit path:
```powershell
& "C:\Program Files\LibreOffice\program\python.exe" script.py
```

**CI:** `actions/setup-python@v5` with `python-version: '3.11'` is used in `.github/workflows/deploy.yml`. Linux runners don't have this stub issue.

## 3. Python 3.8 doesn't support `tuple[Number, Number]` builtin generic syntax

**Symptom:** `TypeError: 'type' object is not subscriptable` on a line like `def f() -> tuple[float, float]:`

**Cause:** Builtin generic syntax for `tuple`, `list`, `dict` was added in PEP 585 (Python 3.9). LibreOffice's Python is 3.8.10.

**Fix:** Use `Tuple` from `typing`:
```python
from typing import Tuple
def f() -> Tuple[float, float]: ...
```
Or use `from __future__ import annotations`.

## 4. Jekyll `webrick` not found

**Symptom:** `bundle exec jekyll build` errors with `require: cannot load such file -- webrick (LoadError)`

**Fix:** `bundle add webrick` (one-time per checkout). The Gemfile in this repo already includes `gem "webrick"`.

## 5. GPG signing fails: "No secret key"

**Symptom:** Local `gpg --armor --detach-sign` returns "No secret key" even after `gpg --list-secret-keys` shows nothing.

**Cause:** By design. Local keyring was DELETED after the private key was uploaded to the GitHub Secret `GPG_PRIVATE_KEY`. Local machines do not sign — GitHub Actions does.

**Fix:** Do not attempt local signing of `_site/index.html` for deploy. To verify a locally-built `_site/index.html`, use the `signature` file fetched from the live site, not a locally-generated one.

## 6. `git push` fails with "Authentication failed"

**Symptom:** `git push origin main` returns 401 even with user.name and user.email set in config.

**Cause:** HTTPS push requires a PAT (Personal Access Token), not the password. The PAT is stored in `.env` (gitignored).

**Fix:**
```powershell
$PAT = (Get-Content .env | Select-String "^GITHUB_PAT=").Line -replace "GITHUB_PAT=", ""
git remote set-url origin "https://open-evidence-ar:$PAT@github.com/open-evidence-ar/rotavirus-argentina.git"
git push origin main
git remote set-url origin "https://github.com/open-evidence-ar/rotavirus-argentina.git"
```

Always restore the URL without the PAT after push.

## 7. METH-006 still WARN after creating public.pem

**Symptom:** `validate_ci.py` reports METH-006 as WARN even though `public.pem` exists.

**Cause:** The METH-006 check inspects `public.pem` for placeholder text. If the file still has placeholder text, the check fails.

**Fix:** Re-export with `gpg --armor --export "Rotavirus Argentina Evidence" --output public.pem` and verify fingerprint matches `93604ADBAFBDBB56D60B37CA8585492BFC14A09A`.

## 8. `sections/*.md` files not being included in the build

**Symptom:** Jekyll builds successfully but `_site/index.html` is missing content from one or more sections.

**Cause:** `index.md` uses `{% include_relative sections/NN-name.md %}`. If the section filename changes, the include breaks silently (becomes empty).

**Fix:** Verify section filenames match the include paths in `index.md`. Run `bundle exec jekyll build --verbose --profile` to see include traces.

---

## 9. `archive.zip` missing from `_site/` after local build

**Symptom:** Footer link to `archive.zip` 404s on live site.

**Cause:** The `archive.zip` is generated by the `Generate archive` step in `.github/workflows/deploy.yml`, not by the local Jekyll build.

**Fix:** Run manually if testing locally:
```powershell
cd _site
Compress-Archive -Path public.pem, integrity.txt, signature, llms.txt -DestinationPath archive.zip
```

## 10. Push fails because of "refusing to allow an integration to create or update code"

**Symptom:** GitHub returns 403 "refusing to allow an integration to create or update code without permission" on push.

**Cause:** The PAT used does not have `repo:write` scope.

**Fix:** Regenerate the PAT at https://github.com/settings/tokens with `repo` scope. Update `.env` accordingly.

## 11. `validate_calculations.py` reports a numeric regression

**Symptom:** A previously-PASS test case now reports `[FAIL]` with `actual=X.X expected=Y.Y |d|=Z`.

**Cause:** A number in `sections/*.md` was changed without updating the test case (or vice versa), OR a primary input value changed.

**Fix:**
1. Check `git diff sections/*.md` for the changed number
2. Check `git diff scripts/validate_calculations.py` for the changed test case
3. Decide which is the source of truth (the test case is the canonical formula; the section text should match)
4. Update the stale side
5. Re-run `python scripts/validate_calculations.py --verbose`

Do NOT loosen the tolerance to make the test pass; investigate first.

## 12. GitHub Actions deploy workflow fails on "Sign build"

**Symptom:** The `Sign build` step in `.github/workflows/deploy.yml` fails with `gpg: no default secret key`.

**Cause:** The `GPG_PRIVATE_KEY` GitHub Secret is empty, malformed, or the key identity string in the workflow step doesn't match the imported key's UID.

**Fix:**
1. Verify Settings → Secrets → `GPG_PRIVATE_KEY` is set with the full `-----BEGIN PGP PRIVATE KEY BLOCK----- ... -----END PGP PRIVATE KEY BLOCK-----` block
2. Verify the workflow step uses `--default-key "Rotavirus Argentina Evidence"` (must match the key UID)
3. Verify the `if: env.GPG_PRIVATE_KEY != ''` condition evaluates true
